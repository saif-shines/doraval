`dora skill lint` catches what schema validation can't: vague language, self-contradicting instructions, unclear triggers, and scope creep. One LLM call, no sessions required.

```bash
dora skill lint <path>
```

`validate` checks structure (frontmatter, required fields, file layout). `judge` checks rubric alignment (does it have examples, guardrails, imperative voice). `lint` reads the skill the way an agent would and flags what's actually confusing.

## Usage

```bash
dora skill lint ./skills/my-skill/
dora skill lint .                          # defaults to current directory
dora skill lint ./skills/my-skill/ --for cursor   # platform-aware linting
```

`<path>` is a skill directory (must contain `SKILL.md`).

## Categories

Each finding is tagged with one of five categories:

| Category | Flags |
|---|---|
| **clarity** | Ambiguous or hedging language a reader could misread |
| **actionability** | Steps that are too vague to execute |
| **contradiction** | Instructions that conflict with each other |
| **trigger** | Missing or weak activation phrasing |
| **scope** | The skill doing more (or less) than its description promises |

Findings are omitted for categories with no issues — `lint` does not invent problems to fill a template.

## Severity and overall verdict

| Severity | Meaning |
|---|---|
| `error` | Blocks reliable use — fix before shipping |
| `warning` | Works but will confuse some agents/readers |
| `info` | Minor polish suggestion |

`overall` is derived programmatically: `fail` if any `error`, `warn` if any `warning` (no errors), `pass` otherwise.

## Example output

```
  judge: claude CLI (no API key configured)

  [WARN]  Mostly clear, one ambiguous trigger and a scope mismatch.
  via: claude CLI

  ⚠ trigger  Description says "use for reviews" but body only handles PR diffs
    → Narrow the description to "use when reviewing a pull request diff"
  ℹ clarity  "you might want to check tests" reads as optional
    → Replace with a direct imperative: "Check tests before merging"
```

A fully clean skill prints:

```
  [PASS]  No issues found.
```

## Options

| Flag | Short | Description |
|---|---|---|
| `--for <platform>` | | Platform-aware linting: `claude` \| `cursor` \| `codex` \| `copilot` |
| `--format <type>` | `-f` | `table` (default) \| `json` |
| `--ci` | | Exit 1 if overall is `warn` or `fail` |

## JSON output

```bash
dora skill lint ./skills/my-skill/ --format json
```

Returns:

```json
{
  "overall": "warn",
  "summary": "Mostly clear, one ambiguous trigger and a scope mismatch.",
  "findings": [
    {
      "severity": "warning",
      "category": "trigger",
      "finding": "Description says \"use for reviews\" but body only handles PR diffs",
      "suggestion": "Narrow the description to \"use when reviewing a pull request diff\""
    }
  ],
  "method": "claude-cli"
}
```

## Configuration

Uses the same judge LLM as `judge` and `drift`:

```bash
dora evals setup     # configure vendor + model
dora init             # full one-time setup (includes evals)
```

Falls back to your local `claude` CLI if no API key is configured. See [evals setup](/commands/evals-setup/) for supported vendors.

## Exit codes

- `0`: success (or `--ci` not set and overall is `warn`/`fail`)
- `1`: with `--ci`, overall is `warn` or `fail`; or skill/judge not available

## `lint` vs `validate` vs `judge`

|  | `validate` | `judge` | `lint` |
|--|---|---|---|
| Checks | Structure/schema | Rubric alignment | Clarity/contradictions |
| LLM required? | No | Yes | Yes |
| Question answered | Is the file well-formed? | Does it meet best practices? | Would an agent misread this? |

Run all three before shipping a skill to a team or community.
