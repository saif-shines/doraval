`dora drift` reads actual coding agent session transcripts, identifies which skills were invoked, and uses an LLM judge to determine whether the agent's **behavior** aligned with the skill's instructions.

Unlike `validate` (static) and `judge` (artifact quality), drift answers: **Did the agent actually follow this skill in a real session?**

## Three modes

### Mode 1 — Single skill, all sessions

```bash
dora drift <skill-path>
```

Scans all recent sessions for the current directory, filters to those that invoked the skill, and runs behavioral analysis on each.

### Mode 2 — Single skill, single session

```bash
dora drift <skill-path> --session <session-id-prefix>
```

Filters to the one session whose path contains the given id prefix.

### Mode 3 — Repo sweep

```bash
dora drift
```

No path = scan all discovered skills in the current directory, run Mode 1 for each, rank by drift rate (highest first).

## Verdicts

Each skill instruction gets one of four verdicts:

| Symbol | Verdict | Meaning |
|--------|---------|---------|
| `✓` | **ALIGNED** | Agent followed the instruction |
| `↗` | **DRIFTED** | Agent did not follow the instruction (with evidence) |
| `~` | **JUSTIFIED** | Agent deviated but provided a valid reason |
| `?` | **UNCLEAR** | Insufficient evidence to judge either way |

`UNCLEAR` is treated as `ALIGNED` (benefit of the doubt) and surfaced as a skill-quality flag — a signal the instruction needs tightening.

## Instruction bindingness

Before judging, the LLM classifies each instruction:

| Bindingness | Meaning |
|-------------|---------|
| **MANDATORY** | Agent must do this — always judged |
| **CONDITIONAL** | Context-dependent — judged when condition applies |
| **DISCRETIONARY** | Example or suggestion — never causes DRIFTED |

Only MANDATORY and triggered-CONDITIONAL instructions count toward the drift rate. This prevents false positives: if a skill says "check 4 repos" and the agent checked 3 because only 3 existed, the agent can be JUSTIFIED rather than DRIFTED.

## Example

```
  dora drift — session-grounded behavioral analysis

  Skill: brainstorming   Sessions checked: 5   Sessions matched: 3

  Session a1b2c3 "Design a new feature"
    ✓ ALIGNED     Invoke the skill before responding    (tool[0]: Skill brainstorming)
    ↗ DRIFTED     Run git log for churn signal          (no git log call found)
    ~ JUSTIFIED   Check 4 repos                         (agent: only 2 repos match scope)
    ? UNCLEAR     Ask clarifying questions first        (instruction too vague to judge)
    2 ALIGNED, 1 DRIFTED, 1 JUSTIFIED, 1 UNCLEAR

  ...

  Aggregate drift rate: 1/8 binding instructions drifted (12%)

  Skill ambiguity flags (UNCLEAR → treated as ALIGNED):
    • "Ask clarifying questions first" — instruction too vague, tighten it
```

Default output (without `--verbose`) shows only DRIFTED and UNCLEAR items. A fully-aligned session shows:

```
    ✓ All instructions aligned
    4 ALIGNED, 0 DRIFTED, 0 JUSTIFIED, 0 UNCLEAR
```

## Options

| Flag | Short | Description |
|------|-------|-------------|
| `--session <id>` | | Filter to the session whose path contains this id prefix (Mode 2) |
| `--format <type>` | `-f` | `table` (default) or `json` |
| `--limit <n>` | | Max sessions to check per skill (default: `20`) |
| `--verbose` | `-v` | Show all checklist items, not just DRIFTED and UNCLEAR |
| `--ci` | | Exit 1 if any DRIFTED items found |

## JSON output

```bash
dora drift ./skills/my-skill/ --format json
```

Returns:

```json
{
  "skill": "brainstorming",
  "sessionsChecked": 5,
  "sessionsMatched": 3,
  "results": [
    {
      "sessionId": "a1b2c3d4-...",
      "sessionTitle": "Design a new feature",
      "verdict": "FAIL",
      "verdictReason": "Agent skipped git log step.",
      "checklist": [
        {
          "instruction": "Run git log for churn signal",
          "bindingness": "MANDATORY",
          "itemVerdict": "DRIFTED",
          "evidence": "no git log call found in tool sequence"
        }
      ],
      "ambiguityFlags": ["Ask clarifying questions first"]
    }
  ]
}
```

Repo sweep (`dora drift --format json`) returns an array ranked by `driftRatePct` (highest first).

## Ambiguity flags

`UNCLEAR` items are aggregated into an `ambiguityFlags` list at the end of output. These are not failures — they receive the benefit of the doubt — but they signal that the instruction needs tightening in the skill before the next session.

## How drift works

1. Discovers recent sessions via the active agent adapter (Claude Code best supported).
2. Parses each session's JSONL to extract:
   - Which skills were invoked (Skill tool calls, `<command-name>` tags, hook content, `attributionSkill`)
   - Ordered tool call sequence with indexes
   - Assistant reasoning text (the "why" written before tool calls)
3. Filters to sessions that invoked the target skill.
4. For each matching session, sends skill content + session data to an LLM judge:
   - Classifies each instruction as MANDATORY / CONDITIONAL / DISCRETIONARY
   - Judges behavior against MANDATORY + triggered-CONDITIONAL only
   - Requires evidence citation per verdict (tool-call index or agent quote)
5. Derives verdict programmatically — only non-DISCRETIONARY DRIFTED items cause FAIL.

## Aliases

```bash
dora drift ./skills/brainstorming/
dora skill drift ./skills/brainstorming/   # same command
```

## Configuration

Requires a judge LLM:

```bash
dora init            # one-time setup
dora evals setup     # re-run just the judge step
```

See [evals setup](/commands/evals-setup/) for supported vendors and config keys.

## Exit codes

- `0`: success
- `1`: with `--ci`, at least one DRIFTED item found
- `2`: fatal (no agent adapter detected, no sessions found)

## When to use drift

- After real agent sessions to measure how well a skill is being followed in practice.
- To identify which instructions are being skipped vs. which are just ambiguous.
- To surface skill-authoring improvements via ambiguity flags.
- In CI against a golden session with `--ci` for critical skills.

`validate` and `judge` are fast and session-free. `drift` is slower and requires sessions but gives the highest-signal behavioral feedback.
