`dora judge` evaluates a skill artifact against a best-practices rubric using a single LLM call. No sessions required — point it at a skill directory and get a verdict immediately.

```bash
dora judge <skill-path>
dora eval  <skill-path>   # alias — identical behavior
dora evals <skill-path>   # alias — identical behavior
```

## Usage

```bash
dora judge <path> [options]
```

`<path>` is the skill directory (must contain `SKILL.md`).

### Default rubric

Judges against the built-in Claude skill quality rubric:

```bash
dora judge ./skills/brainstorming/
```

### Platform rubric

```bash
dora judge ./skills/my-skill/ --for codex
dora judge ./skills/my-skill/ --for cursor
dora judge ./skills/my-skill/ --for copilot
```

### Custom rubric

```bash
dora judge ./skills/my-skill/ --rubric ./my-best-practices.md
```

Loads any markdown file as the rubric. Useful for team-specific standards.

## Example output

```
  [PASS] Rubric alignment
  All mandatory criteria met: activation phrase, ordered steps, imperative voice,
  code examples, and explicit MUST / MUST NOT guardrails.
```

With `--verbose`:

```
  [FAIL] Rubric alignment
  Two mandatory criteria DRIFTED — no code examples and no guardrails.

  Checklist:
    ✓  [MANDATORY] Activation phrase present
    ✓  [MANDATORY] Step-by-step structure
    ✓  [MANDATORY] Imperative voice
    ✗  [MANDATORY] Code examples in body
         evidence: no fenced code blocks found
    ✗  [MANDATORY] Explicit MUST / MUST NOT guardrails
         evidence: no MUST or MUST NOT in body
```

## Verdicts

Each rubric criterion gets one of four verdicts:

| Symbol | Verdict | Meaning |
|--------|---------|---------|
| `✓` | **ALIGNED** | Criterion satisfied |
| `✗` | **DRIFTED** | Criterion violated (with evidence) |
| `~` | **JUSTIFIED** | Deviation with valid reason |
| `?` | **UNCLEAR** | Insufficient evidence to judge |

Top-level verdict: `PASS` if no MANDATORY or CONDITIONAL criterion is DRIFTED, `FAIL` otherwise. The verdict is derived programmatically — the LLM cannot override it. `UNCLEAR` items receive the benefit of the doubt (treated as ALIGNED) and are surfaced in `ambiguityFlags`.

## Signed verdict envelope (JSON output)

```bash
dora judge ./skills/brainstorming/ --format json
```

Returns:

```json
{
  "verdict": "PASS",
  "verdictReason": "All mandatory rubric criteria met.",
  "rubricRef": "built-in:claude",
  "model": "glm-4-flash",
  "provider": "zai",
  "judgeMethod": "api",
  "timestamp": "2026-07-01T10:00:00.000Z",
  "doravalVersion": "0.4.1",
  "checklist": [
    {
      "instruction": "Activation phrase present",
      "bindingness": "MANDATORY",
      "itemVerdict": "ALIGNED",
      "evidence": "when_to_use: \"Use when user invokes brainstorm\"",
      "detail": null
    }
  ],
  "ambiguityFlags": []
}
```

`rubricRef` is either `built-in:<platform>` or the resolved path to the custom rubric file. The envelope records which model + provider produced the verdict so results are auditable over time.

## Options

| Flag | Short | Description |
|------|-------|-------------|
| `--rubric <file>` | | Path to custom best-practices markdown file |
| `--for <platform>` | | Built-in rubric: `claude` (default) \| `codex` \| `cursor` \| `copilot` |
| `--format <type>` | `-f` | `table` (default) \| `json` |
| `--verbose` | `-v` | Show full checklist (default: summary only) |
| `--ci` | | Exit 1 if verdict is FAIL |

## Configuration

Requires a judge LLM to be configured:

```bash
dora evals setup
```

See [evals setup](/commands/evals-setup/) for the interactive flow and supported vendors. Same configuration applies to `drift`.

## Exit codes

- `0`: success (PASS or informational run)
- `1`: with `--ci`, verdict is FAIL; or skill/rubric file not found
- `1`: judge LLM not configured (run `dora evals setup`)

## judge vs drift

|  | `judge` | `drift` |
|--|---------|---------|
| Sessions required? | No | Yes |
| LLM calls | 1 | 1 per session |
| Speed | Fast | Slower |
| Question answered | Is the skill well-authored? | Did the agent follow the skill? |

Use `judge` when writing or reviewing a skill. Use [`drift`](/commands/drift/) after real sessions to measure behavioral adherence.
