`dora fix` applies mechanical fixes from scan and review findings. It always shows diffs and asks before writing unless you pass `--yes`. Use it after `dora` or `dora review` when the report lists auto-fixable issues.

```bash
dora fix [PATH]
```

## Mechanical vs judgment

| Kind | Examples | What `fix` does |
| --- | --- | --- |
| Mechanical | Missing frontmatter fields, path mismatches, renames | Shows a unified diff; applies on approval |
| Judgment | Vague descriptions, missing guardrails | Does not auto-write. Use `--brief` for an agent-ready prompt, then re-run `dora review` |

## Options

| Flag | Meaning |
| --- | --- |
| `--yes` | Pre-approve all fixes (agents / CI) |
| `--dry-run` | Show diffs; write nothing |
| `--brief` | Emit an agent-ready prompt for judgment fixes |
| `--format table\|json` | Output format |
| `--ci` | Machine mode (implies JSON) |
| `--cwd <path>` | Working directory override |

## Examples

```bash
dora fix
dora fix ./skills/deploy
dora fix --dry-run
dora fix --yes
dora fix --brief
```

## Common mistakes

- Running `fix` with no prior scan/review: there may be nothing to apply. Run `dora` or `dora review` first.
- Expecting judgment items to rewrite themselves: use `--brief` and your coding agent, then re-review.
