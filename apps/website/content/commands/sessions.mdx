`dora sessions` lists recent agent sessions for the current project and shows a timeline for one session. It is read-only. Use it when you want human-readable history, or to understand what `dora review --sessions` mines for skill-usage evidence.

```bash
dora sessions
dora sessions --agent claude --limit 20
dora sessions show <id>
```

Adapters ship for Claude Code, Grok, Cursor, Codex, and Copilot, so sessions from each surface appear here when logs exist on disk.

## Options (list)

| Flag | Meaning |
| --- | --- |
| `--agent <name>` | Filter (`claude`, `grok`, `cursor`, `codex`, `copilot`) |
| `--limit <n>` | Max sessions per agent (default `10`) |
| `--format` / `--ci` | Machine output |

## show

Timeline: user turns, tool calls (names + args summary), skills invoked, token totals when available.

```bash
dora sessions show <sessionId>
```

## How review uses sessions

The session tier of `dora review` mines these logs for mechanical evidence that a skill was actually invoked (not only that the file exists). `--sessions` requires that tier and exits `2` if no recent sessions exist.

## Common mistakes

- Empty list: agents write logs only after real runs; new machines or sandboxes often have none yet.
- Treating `sessions` as a chat UI: it summarizes local logs for diagnosis, not live conversation.
